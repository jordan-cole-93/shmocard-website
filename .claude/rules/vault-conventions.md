## What's in the vault

- `context.md` — project context (business, thesis, sub-brands)
- `marketing.md` — voice, copy, headlines, angles, proof points
- `product.md` — product catalog, formats, status
- `wireframe/sitemap.md` — full site structure
- `wireframe/home-page-shmocard.md` — homepage wireframe
- Daily logs, meeting notes, ad-hoc captures

The vault is **separate** from this repo. Don't duplicate vault notes into the repo.

## Hard rules

1. **Ask before editing existing vault notes.** Jordan curates his vault carefully. Modifying his notes without permission is intrusive and risks overwriting nuance.
2. **`(AICHECK)` prefix on AI-created files.** Any file Claude creates in the vault gets the prefix `(AICHECK)` so Jordan knows what's machine-generated and can review it.
3. **Read freely.** Read access to the vault is fine for context. The constraint is on **writes** — modifications and new files.
4. **Use vault-sync skill for capture.** When Jordan asks to "save this to my vault," "capture this," or "add this to Obsidian," use the `vault-sync` skill — don't write to the vault directly via raw file tools.
5. **Don't move vault notes into the repo.** Vault notes are personal context; repo is shareable code. Keep them separate.

## When you need vault info

- **Read** the relevant vault note directly (paths above).
- **Reference** vault notes in repo files using their full path (so Jordan can click through).
- **Don't paste** large vault excerpts into repo files — link to them instead.

## Why this matters

The vault is Jordan's working memory across all his businesses (Shmocard, Pawn Leads, Cole Digital, Walceo). Sloppy edits to vault notes cause cross-project confusion and break his daily review flow.
